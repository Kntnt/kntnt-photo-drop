# Slideshow as a passive fullscreen surface with a pluggable trigger contract

The Gallery block gains a **Slideshow**: a visitor-started, automatically advancing, endlessly looping fullscreen playback of the gallery's view — exactly the images the block renders, in the block's order, the same set the lightbox pages through. It is a third surface alongside the gallery grid and the lightbox, fully orthogonal to the lightbox/download click matrix (ADR-0007), and it is deliberately **passive**: the only interaction is ending it, via ESC, the browser's native fullscreen exit, or a discreet always-visible close button (the touch path). The lightbox remains the browsing surface; the slideshow is the lean-back surface. Fullscreen uses the Fullscreen API where available, with a viewport-filling overlay fallback where it is not (notably iPhone Safari); both paths end the slideshow the same way and return the visitor to the gallery.

The trigger is a three-state block choice rather than a toggle: **off** (default), **built-in button** (a quiet affordance above the gallery whose label the editor sets, since the label is per-page content), or **custom element** — the page designer places any element anywhere on the page carrying `data-kntnt-photo-drop-slideshow`, whose value names the target gallery's HTML anchor (core's `supports.anchor` field); a valueless attribute targets the first slideshow-enabled gallery on the page, so the common one-gallery page needs no anchor at all. The data attribute is a documented, public contract with page builders.

Per-slide duration is a block attribute (integer seconds, default 5, minimum 1) measuring the time a slide stands fully visible — the transition comes on top. The transition is a hardcoded ~1 s dissolve with no attribute until a second transition actually exists (the pre-1.0 no-BC policy makes adding the enum later free); `prefers-reduced-motion` replaces the dissolve with a hard cut. The slideshow never transitions to an image that has not finished loading — the next slide preloads during display, and a late image extends the current slide instead of dissolving to a blank. A screen wake lock is held while playing (silently degrading where the API is missing), since endless playback implies photo-frame/kiosk use. The slides carry the gallery's shared caption overlay under the existing Caption settings, exactly as the lightbox does; no counter and no download icon appear during playback.

## Considered Options

- **An autoplay mode inside the lightbox**: rejected — it blurs the two surfaces' roles, and the lightbox's chrome (counter, paging controls, download icon) is precisely what the slideshow must not show.
- **An anchored icon overlay on the gallery container** (the download icon's nine-point vocabulary): rejected — four more bespoke styling attributes and poor discoverability, since the icon would have to be understood as applying to the whole gallery rather than an image.
- **A play button in the lightbox chrome**: rejected — it would couple the slideshow to lightbox-on, and the two are independent choices.
- **Built-in button only, no custom trigger**: rejected — page designers need placement freedom; the documented data-attribute contract buys it at the cost of a public HTML contract, which this ADR records.
- **Always requiring the anchor on the custom trigger**: rejected — needless friction on the common single-gallery page; the forgiving first-match rule is documented behaviour, not an accident.

## Consequences

The view module scans the document for foreign trigger elements — surprising without this ADR, and the reason the attribute is namespaced `data-kntnt-photo-drop-*`. Once shipped, the attribute name and its anchor-targeting semantics are a contract with content; changing them post-1.0 is a breaking change to pages, not just to code. The playback set is the *gallery's view*, never the whole collection, so no new data source exists — the slideshow reuses the gallery's rendered list, order, captions, and responsive srcsets. Playback is visitor-started and stoppable at all times, which keeps the auto-advancing content inside WCAG's expectations without a pause control.

# Drop Zone upload controls are token-wired core links, not server chrome

The Drop Zone's visible upload controls — the "Add photos" trigger and the folder picker — are no longer server-emitted chrome. They are ordinary blocks the site builder authors inside the Drop Zone (a `core/button`, a link, anything carrying an `href`), wired to the uploader by an **anchor-token href**: a link whose target is `#kntnt-drop-zone-files` opens the loose-file picker, and one whose target is `#kntnt-drop-zone-folder` opens the `webkitdirectory` folder picker. This is the same idea as the `{kntnt-drop-zone-collection}` placeholder (the collection-name substitution `render.php` already performs), but it lives in the href instead of in text — so the token must be a **valid URL**, a `#fragment`, because the block editor's link control and `esc_url()` strip the braces a `{…}` token would need and the round-trip would mangle it. The default inner-block template seeds a `core/buttons` carrying the two tokened buttons, so a freshly inserted Drop Zone looks and behaves exactly as before; the template is unlocked, so the builder relabels, restyles, repositions, or removes either control, or turns the folder one back into a quiet text link. `render.php` keeps every server responsibility that is **not** presentation: the capability gate (a visitor without `upload_files` still gets nothing and no nonce, ADR-0006), the collection-name replacement, the two **hidden** file inputs the controls trigger, and the live summary plus per-file status list (dynamic regions the view module owns, never editable content). The view module finds the tokened links by their href and wires each click to the matching hidden input; with the controls now part of `post_content`, a builder can delete them — including the only keyboard-reachable upload path — which is the builder's responsibility, like any block they choose to remove.

## Considered Options

- **Two dedicated Drop-Zone-only child blocks** (`upload-button`, `folder-picker`, restricted by `parent`): rejected — a block type insertable only inside one parent is a heavy, single-use addition to the inserter for what a plain core button plus one convention already does, and the maintainer explicitly did not want blocks that exist solely to live in the Drop Zone.
- **Inspector fields on the Drop Zone for the two labels:** rejected as the primary mechanism — it makes the *labels* editable but not the *controls* (no restyle, no reposition, not authored content), and the request was specifically for editable inner blocks. It survives only as the fallback had the token idea failed.
- **A brace token `{kntnt-drop-zone-files}` in the href**, mirroring the collection placeholder verbatim: rejected — `{` and `}` are not URL characters, so the editor's link control and `esc_url()` strip or percent-encode them and the token never reaches `render.php` intact. The `#fragment` form is the URL-safe realisation of the identical idea, and it degrades gracefully without JavaScript (a fragment to a non-existent id simply does nothing).
- **Server-rewriting the token into a `data-` attribute with an inert href:** rejected as unnecessary — a fragment to a missing id is already inert (no navigation, no scroll), so leaving it intact lets the view module key off the author's own href without the render handler reaching into builder markup beyond the single collection-name substitution it already performs.

## Consequences

This supersedes the part of [ADR-0008](0008-ingestion-placement-hierarchy-and-uploader-folders.md)'s native-surface amendment that made the folder picker a *we-controlled* "quiet inline link": its emphasis, label, and even its element type are now the builder's, not the plugin's. The folder picker also loses its native `<label>`-association trigger; like the "Add photos" control it becomes a link whose click calls the hidden input's `.click()` — the identical, already-accessible pattern the loose-file button used, so keyboard and touch users keep a working folder route as long as the control is present. Because the controls live in `post_content`, the accessible upload paths are now deletable: a builder who removes both buttons leaves only the pointer-only click-anywhere surface, which is why the seeded template keeps them and why this trade is recorded rather than hidden. Nothing about the capability gate, the server-enforced contract, or the uploader-folder placement changes — those stay in `render.php` and the REST endpoint ([ADR-0006](0006-server-enforced-contract-rest-upload.md), [ADR-0008](0008-ingestion-placement-hierarchy-and-uploader-folders.md)). Per the pre-1.0 no-backwards-compatibility policy, no migration is provided: a Drop Zone saved before this change carries no controls in its inner blocks (it relied on the now-removed server chrome), so the builder re-inserts the block to get the seeded buttons, or adds a tokened link by hand.

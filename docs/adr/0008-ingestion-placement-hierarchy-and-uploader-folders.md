# Dropped folders preserve hierarchy; an optional per-uploader namespace

Two ingestion-placement decisions are settled here. **First, a dropped folder preserves its hierarchy exactly like the "Select folder" picker.** A folder dragged onto the Drop Zone is traversed recursively and every file's source-relative path is recreated under the collection, identical to what `webkitRelativePath` carries for the `webkitdirectory` picker — a drop and the picker now have one semantics, and the earlier detect-warn-flatten consent flow disappears. The flatten rule was a FilePond-era constraint; the native drop surface (ADR-0006 amendment, v0.4.0) removed the reason for it, and preserving the tree is lossless. **Second, the collection descriptor gains a boolean `uploader_folders` (default `true`, fixed at establishment).** When it is on, every Drop Zone upload is placed under a first path segment derived **server-side** from the authenticated uploader's `user_nicename`, ahead of the upload's own relative path — an **uploader folder**. With the filesystem as the only source of truth (ADR-0001), the path is the only place uploader provenance can live: `index.json` is a regenerable cache (ADR-0003), EXIF is stripped by the ingestion pipeline (ADR-0006), and database rows are forbidden. The prefix doubles as a collision firewall — two photographers who both drop `IMG_0001.webp` no longer overwrite each other. This is an ingestion-placement rule for the **Drop Zone REST path only**: CLI `image import` is unaffected, and the doctor neither enforces nor knows about it — nothing becomes a foreign file because of this option.

## Considered Options

- **Keep the per-drop detect-warn-flatten consent flow** (the original contract: a dropped folder is warned about and, on consent, contributes only its top-level files, flat): rejected — it was a FilePond-era workaround, it gives a drop and the picker *different* results for the same input, and the native drop surface can traverse a tree recursively, so the flatten is now a pointless loss.
- **A per-collection "flat vs hierarchical" layout flag:** rejected — YAGNI. Preserving the hierarchy is lossless, and the Gallery already recursive-flattens for display (ADR-0005), so a stored flatten would discard structure the viewer never needed flattened on disk in the first place.
- **A block-level setting for the uploader namespace** (a Drop Zone attribute toggling the prefix): rejected — the prefix is a property of *where images land*, which the collection owns; the Drop Zone is a select-only consumer (ADR-0001) and must not be able to change a collection's on-disk placement rule. Two Drop Zone blocks pointing at one collection must agree, and only the descriptor can make them agree.
- **A mutable option** (changeable after establishment, like the display name): rejected — flipping it later would split one uploader's images across two layouts (prefixed and bare) with no migration path, since the filesystem is the truth and there is no index to rewrite. Fixing it at establishment, alongside the output contract, keeps every image in a collection placed by one rule.
- **`user_login` instead of `user_nicename` for the segment:** rejected — the login name is sensitive (it is half of a credential pair and is never meant to be public), whereas the nicename is WordPress's already-public author slug. A path segment is web-visible, so it must be built from the public identifier, not the private one.

## Consequences

The trade-off is that **nicenames appear in public image URLs** when `uploader_folders` is on. This is mild: WordPress already exposes the same nicename in every author archive URL (`/author/<nicename>/`), so the option reveals nothing the site does not already publish, and a site that finds even that unacceptable simply leaves the option off at establishment. Because the segment is derived server-side from the authenticated user at upload time, the client never names it and cannot spoof it; it is prepended before the per-file relative path is hard-sanitised and `realpath`-confined to the collection root (ADR-0006), so the namespace cannot be escaped. The doctor stays neutral — an uploader folder is an ordinary content folder, its mains and derived artifacts reconciled like any other, and nothing about this option can produce a foreign file. CLI `image import` writes where it is told, with no prefix, because it runs in a trusted context with no authenticated uploader to derive a nicename from; the option is meaningful only on the Drop Zone path that *has* an authenticated uploader.

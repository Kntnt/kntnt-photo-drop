# Slideshow re-syncs to the gallery's current view at each cycle boundary, by refetching the page

The Slideshow becomes live: at each **cycle boundary** — the moment the last slide has stood its visible time and playback is about to restart from the first — the controller fetches the page's own URL (`cache: 'no-store'`, same-origin credentials), parses the response with `DOMParser`, locates the same gallery wrapper (by HTML anchor id when one is set, otherwise by document-order index among gallery wrappers), reads its thumbnail anchors through the existing `readSlides()` data contract, and **replaces its slide list wholesale**. The resync is full: additions, deletions, caption and srcset changes, and ordering all follow the server's current view, so every cycle plays exactly what a page reload would show at that moment. This amends ADR-0009's consequence that "no new data source exists": the data *contract* is unchanged — the gallery's rendered anchors — but a fresh copy of the page is fetched once per cycle. The motivating scenario is the photo frame ADR-0009 already anticipated: a slideshow started early in an event keeps rolling for hours while the photographer's uploads stream in through the Drop Zone, and each new cycle picks them up.

The wrap waits for the sync, bounded. The fetch starts when the last slide begins its visible phase; if it has not settled when the slide's time is up, the wrap waits for it — the same philosophy as the advance gate, where a slow image extends the current slide rather than dissolving to a blank — bounded by a short abort timeout. On a **transport failure** — timeout, network error, or a non-OK HTTP status — the stale list wraps and the next boundary retries; persistent failure thus degrades to exactly the pre-0011 behaviour. An **empty fresh view** ends the playback at the boundary: deletion is often takedown-motivated, and a takedown must propagate to a long-running frame within one cycle. A successfully fetched page that would show no images **is** the empty view, in either of its two markup shapes: `Render_Gallery` renders a *deleted or otherwise broken* collection as **no wrapper at all** for the public (a missing wrapper), and an *emptied but still-present* collection as the **empty-message wrapper carrying no slide anchors** ([ADR-0012](0012-imageless-gallery-shows-a-public-message.md)). Both carry the same takedown meaning ("a reload would show no images") — a missing wrapper and a wrapper with zero anchors each resolve to an empty slide list — so both end the playback. Only transport-level failures keep the stale list; a flaky network can therefore never kill a frame, and a takedown can never be mistaken for flakiness.

A single-image gallery gets a cycle boundary too: its lone slide's visible time defines the cycle, so it resyncs every `slideshowSeconds` — a full page fetch every few seconds, accepted because the state is the transient bootstrap of the photo-frame case (the cycle lengthens as soon as more images land) and the load is one visitor's device. An empty gallery remains unstartable (no anchors, no mounted controller, no revealed button), exactly as today. The resync is unconditional — no block attribute, no inspector control — and applies to the slideshow only: the gallery grid and the lightbox stay frozen at the page-load view and follow the page's normal lifecycle, so a visitor ending a long playback may return to a grid older than what they just watched, reconciled on reload.

## Considered Options

- **A public REST endpoint that re-renders a named block instance** (post ID + gallery anchor/index, server reads the attributes out of stored `post_content`, returns the slide list as JSON): rejected — it would be the plugin's first public REST surface and would have to re-implement post visibility (status, password protection, private posts) and block-instance resolution, for no gain beyond a lighter payload.
- **A public REST endpoint taking raw parameters** (collection slug + start path + order): rejected — an enumeration and traversal surface letting anyone list any collection's contents, including collections not exposed on any page; `Gallery_Walker` is deliberately designed with no per-request path input precisely to avoid that surface.
- **Append-only sync** (new images added, deletions and order ignored): rejected — the slideshow would diverge from the gallery's view in both directions, and deleted (possibly takedown-required) images would linger until their URLs started failing.
- **Applying the fresh list mid-cycle** as soon as the response arrives: rejected — it abandons the cycle-boundary semantics and makes position mapping hairy when images were deleted; boundary application keeps the restart position trivially at index 0.
- **A black waiting state** for galleries that are empty at start or become empty mid-playback: rejected — a new controller state and a startable-but-empty visitor experience nobody has asked for; a reload once images exist bootstraps the frame.

## Consequences

The view script fetches its own page during playback — surprising without this ADR, and at most one full-page HTML fetch per cycle (per visible period in the single-image bootstrap case). Freshness inherits the page's cache behaviour: a full-page cache or CDN delays pickup exactly as long as it would delay a manual reload, which is the documented contract ("what a reload would show"), not a bug to fix with cache-busting. The last slide may stand somewhat longer than its configured seconds while a slow sync settles, bounded by the abort timeout. The slideshow can lawfully diverge from the surrounding stale grid mid-session. ADR-0009's core claim — the playback set is the *gallery's view*, never the whole collection — is preserved and in fact strengthened: it is now the gallery's *current* view, not its page-load snapshot.
